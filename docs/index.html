01 — Positioning

From “all-in-one that replaces competitors” to “the intelligence layer that rides the standards”

The single most valuable change to the plan. It shrinks what you have to build and makes “supreme” actually reachable.

The ecosystem standardized while the v3.0 plan was being drafted. Three open standards now define the substrate every coding agent shares:

  • MCP — the universal tool/integration protocol. Governed by the Linux Foundation, adopted by Anthropic, OpenAI, Google and Microsoft, with 10,000+ public servers and ~1B tool calls/month by early 2026. Your bet was right — but it’s now table stakes, not an edge.
  • SKILL.md — the portable, model-invoked capability format. The same file runs across Claude Code, Codex CLI, Cursor, Gemini CLI, Copilot and 15+ other agents. Your “skill layer with YAML frontmatter” already is this standard.
  • AGENTS.md — the universal project-context file (where CLAUDE.md and .cursorrules are agent-specific).

The reframe

Don’t build a proprietary mega-plugin that “replaces” Cursor/Cline/Copilot. Build the orchestration + intelligence layer that emits these open standards and adds what none of them have: role awareness, compliance gating, context-budget engineering, and framework selection. You win by riding the rails everyone already laid.

This is consistent with every architecture decision already made in v2.0/v3.0 — MCP-native, additive skill layer, Markdown + YAML. The only change is the story: PromptWise is a conductor, not a replacement.

03 — New: framework-orchestration layer

BMAD, Spec Kit and Kiro become pluggable backends — not features you rebuild

The agentic-development frameworks you asked about already exist as free, open-source, cross-agent, markdown-based systems. Orchestrate them.

The architecture: a framework router

PromptWise adds one new decision layer that selects and wires the right framework per task, then feeds it through your existing role + compliance + context engines.

# New: framework_router.py (sits beside router.py) task ──▶ framework_router │ classify intent + scale + risk ├─ greenfield feature, regulated ─▶ Spec Kit + constitution(compliance) ├─ full product build, multi-role ─▶ BMAD personas → your 14 roles ├─ brownfield change, gated ───────▶ OpenSpec ├─ PRD → tasks only ───────────────▶ TaskMaster (MCP) └─ save-triggered automation ──────▶ Kiro-style hooks (plugin_system) │ ▼ role_intelligence · compliance_engine · compression_engine │ ▼ emitted as MCP tools + SKILL.md + AGENTS.md

Discipline — don’t become framework #10

Reviewers consistently warn these frameworks are over-engineered and “fight you” when assumptions don’t fit. Your value is reducing the chaos of choosing and configuring among nine of them, not adding another. Wrap the best 2–3; expose the rest as optional backends.

License note (you’re going public)
BMAD is MIT and most SDD tools are permissive OSS — you can wrap and ship them, but you must keep their LICENSE files and attribute them in your README/NOTICE. This is cheap to do and signals credibility to technical reviewers.

04 — New: SDLC & project / product / program management

The artifact chain is your PM/program engine — and your regulated-industry wedge

The roles you wanted to serve (PM, product, program) are exactly what spec-driven frameworks already produce.

Every framework runs the same artifact chain. PromptWise’s job is to keep it traceable, role-aware and compliant:

PHASE 01 · ANALYST

Brainstorm & market reality-check

Idea → validated problem. Maps to your Analyst / C-Suite roles.

PHASE 02 · PRODUCT

PRD

Requirements as the source of truth. Stored in memory_manager (Tier: project).

PHASE 03 · ARCHITECT

Architecture & constitution

Technical design + non-negotiable rules. Constitution = compliance_engine activation point (banking / HIPAA / legal).

PHASE 04 · PM / SCRUM

Epics → user stories → tasks

Program/PM layer. TaskMaster or BMAD scrum-master agent shards work with per-task context.

PHASE 05 · DEV / QA

Implement → review → test

Your QA + ML/AI + technical role packs. Context compressed to control token spend.

PHASE 06 · DELIVER

Commit with traceable history

Versioned artifact chain = audit-defense blueprint for regulators.

Why this is the wedge, not just a feature

The real bottleneck in AI coding isn’t capability — it’s trust: surveys show ~84% of developers use AI tools but only ~33% trust the output. The auditable PRD→architecture→story→commit trail is exactly what regulated buyers (banking, legal, healthcare — your highest-value tier) need to sign off. Your SDLC feature and your enterprise pricing are the same lever.

Cost reality to design for
Spec-driven flows re-read the spec/plan/tasks each turn — budget ~20–40% higher token spend than vibe-coding. Your compression_engine.py turns that liability into a sellable benefit (“30% fewer tokens, same rigor”).

06 — Infrastructure & cost

Your bootstrap math holds

Launch-scale infra is single-digit dollars/month; the cost wall only appears at regulated-enterprise scale.

Paid SaaS scale+ Postgres + managed MCP gatewaylow hundreds / mo

Route around the $50K–150K custom-enterprise-MCP wall by using a managed gateway (Composio / MintMCP / Apigene) for OAuth + compliance rather than building it. Your ~$530 build estimate on Claude Pro remains realistic for the engine itself.

Compliance clock
EU AI Act provisions phase in through August 2026 with penalties up to €35M / 7% of turnover. If you target EU regulated buyers, prioritize audit trails and access controls early — which the SDLC artifact chain already gives you.

07 — Go public, free: GitHub

Host PromptWise on GitHub to showcase your skills

A clean public repo is the strongest portfolio artifact you can have — it shows architecture, product thinking and execution in one place.

  1. Create the repo

    Public repo named promptwise. Add a short, specific description: “Cross-agent intelligence layer for AI coding agents — MCP + SKILL.md, role-aware, compliance-ready.” Add topics: mcp, ai-agents, claude, cursor, skill-md, agentic-ai, llm-tools. Topics drive GitHub discovery.

    $ git init $ git remote add origin https://github.com/<you>/promptwise.git

  2. Make the README the product

    This is what 90% of visitors read. Lead with a one-line value prop, a diagram, a 30-second quickstart, and a “works with” badge row (Claude Code · Cursor · Codex · Gemini). Embed the architecture image from this plan. Keep it scannable.

  3. Add a LICENSE + NOTICE

    Pick MIT or Apache-2.0 for your own code. Add a NOTICE / “Credits” section attributing wrapped OSS (BMAD-MIT, Spec Kit, etc.) with links. This protects you and signals professionalism.

  4. Structure the repo for readers, not just runtime

    promptwise/ ├─ README.md # the pitch ├─ docs/ # this plan, architecture, decisions ├─ src/promptwise_v2/ # the engine ├─ skills/ # portable SKILL.md packs ├─ examples/ # 1–2 runnable demos └─ LICENSE · NOTICE · CONTRIBUTING.md

  5. Publish the plan as a live site (free)

    Drop this HTML into /docs, enable GitHub Pages (Settings → Pages → deploy from /docs). You get a public URL like you.github.io/promptwise to link from LinkedIn — a polished, hosted plan with zero hosting cost.

  6. Cut a tagged release

    Even a v0.1.0 release with notes makes the project look alive and intentional. Releases appear in your profile activity and are linkable.

    $ git tag v0.1.0 && git push --tags

  7. Set up your profile README

    Create a repo named exactly your username → its README shows on your GitHub profile. Pin promptwise. Add a one-line bio framing you as “building cross-agent AI developer infrastructure.”

  8. Optional polish that punches above its weight

    A GitHub Actions badge (even a simple lint/test workflow), a short demo GIF in the README, and a CONTRIBUTING.md. These three signal “real project” more than thousands of lines of code do.

Honesty guardrail
Frame it as building in public. Don’t imply traction you don’t have (stars, users, revenue). “Early-stage, actively building” is credible and respected; inflated claims are the fastest way to lose technical trust.
08 — Go public, free: LinkedIn

Promote it to showcase capability — without spending a rupee

The goal isn’t virality; it’s a credible public trail that shows how you think. Three honest posts beat thirty hype ones.

Strategy in four moves

  • Lead with insight, not announcement. The strongest signal you can send is that you understand why the ecosystem is shifting (open standards), not just that you built a thing.
  • Show the thinking, link the proof. Each post ends with a link to the GitHub repo or the hosted plan — the post sells the idea, the repo proves the execution.
  • Cadence: 1–2 posts/week for 4 weeks. Build-in-public arc: the insight → the architecture → a lesson → a milestone. Consistency beats frequency.
  • Use the Featured section. Pin the repo + hosted plan to your profile’s Featured area so visitors see it without scrolling.

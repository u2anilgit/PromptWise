PromptWise Developer Guide

Architecture · Extension · Testing · Plugin Structure

This guide covers PromptWise internals for contributors and integrators. It explains the architecture, module responsibilities, database schema, and step-by-step instructions for adding new tools, providers, or roles.

System Overview

PromptWise is a Python MCP (Model Context Protocol) server that exposes 13 tools to Claude Code. It runs as a subprocess launched by the Claude Code harness when the plugin is installed.

→

No external calls. PromptWise does not call any LLM or external API. Token counting uses tiktoken locally. All pricing is read from YAML files on disk.

Data Flow: Single Tool Call

1. Claude Code → MCP stdio → server._call_tool(name, arguments)
   name = "route_request"
   arguments = {"text": "Extract entities...", "intent": "extract"}

2. call_tool() dispatches to ctx.router.route(...)
   Router detects intent, stakes, tool-chain depth
   Selects tier → looks up model_id in config.providers
   Counts tokens via tokenizer.count_tokens()
   Returns RouteResult (frozen dataclass)

3. call_tool() records to stats
   await ctx.stats.record(tool="route_request", model=result.recommended_model, ...)
   → async INSERT INTO history

4. call_tool() returns JSON string
   json.dumps({recommended_model, reason, estimated_input_cost_usd, ...})
   → [TextContent(type="text", text=result)]
   → MCP stdio → Claude Code

Dependency Injection Pattern

ServerContext is a simple dataclass that holds all initialized service instances. It is constructed once in main() and passed to every tool handler via closure:

@dataclass
class ServerContext:
    config: AppConfig
    rewriter: Rewriter
    optimizer: Optimizer
    router: Router
    cache_planner: CachePlanner
    batcher: Batcher
    summarizer: Summarizer
    stats: StatsService
    session_manager: SessionManager

# In main():
ctx = ServerContext(
    config=load_config(config_dir),
    rewriter=Rewriter(config),
    ...
)

# Handler closure captures ctx:
@server.call_tool()
async def _call_tool(name, arguments):
    result = await call_tool(ctx, name, arguments)
    ...
Why this pattern? Services are stateless and read AppConfig at construction. reload_config hot-swaps ctx.config, ctx.rewriter, and ctx.optimizer in-place. The closure-based handler avoids global state.

Config Pipeline: YAML → Pydantic → AppConfig

load_config(config_dir) workflow:

pricing.yaml  → _load_yaml() → dict → PricingYAML(**data)
providers.yaml → _load_yaml() → dict → ProvidersYAML(**data)
roles.yaml    → _load_yaml() → dict → RolesYAML(**data)

_validate_provider_tiers(providers, pricing)
  → checks every tier model_id exists in pricing.models

promptwise.yaml (optional) → TimeoutConfig

→ AppConfig(pricing, providers, roles, default_model, last_verified, timeout)
All ConfigError exceptions surface at server startup — before any MCP tool is called. This fail-fast approach prevents runtime surprises from malformed YAML.

Pydantic Config Models

pricing.yaml → PricingYAML
  • schema_version: int — migration guard
  • last_verified: str (ISO date) — staleness check (90-day warning)
  • default_model: str — model ID used when none specified
  • models: dict[str, ModelPricing] — keyed by model ID
ModelPricing fields:
  • display_name, provider, tier
  • context_window, max_output (both validated > 0)
  • rates: RateSpec (7 rate fields, all > 0)
  • tokenizer: TokenizerSpec (family, inflation_vs_baseline)
  • notes: str (optional, shown in compare_providers)
providers.yaml → ProvidersYAML
  • providers: dict[str, ProviderConfig]
  • ProviderConfig: tiers (fast/balanced/powerful → model ID), peak_hours_utc, aliases, feature_warnings
  • aliases validated: lowercase, no spaces
  • peak_hours_utc validated: each value 0–23
roles.yaml → RolesYAML
  • preamble_phrases: list[str] — stripped from prompt start
  • filler_words: list[str] — removed from all positions
  • roles: dict[str, RoleSpec] — each has display_name, prefix, description

Cross-File Validation

_validate_provider_tiers() runs after all three YAML files are loaded. It checks that every model ID referenced in providers.yaml tiers exists as a key in pricing.yaml models:

for provider_name, provider in providers_config.providers.items():
    for tier_name, model_id in provider.tiers.model_dump().items():
        if model_id not in pricing_models:
            raise ConfigError(
                f"providers.{provider_name}.tiers.{tier_name} references '{model_id}' "
                f"which is not in pricing.yaml models"
            )

server.py — MCP Entry Point

Two module-level functions handle all MCP operations. They are module-level (not methods) so tests can call them directly without starting the full server:

  • list_tools(ctx) — returns list[Tool] with JSON schema for each tool's input
  • call_tool(ctx, name, arguments) — dispatches by name, returns JSON string. All errors caught and returned as {"error": ..., "type": ..., "tool": ...}

The main() async function wires everything together:

  1. Resolves config directory as 2 levels above __file__ (project root)
  2. Calls init_db() to apply migrations idempotently
  3. Constructs ServerContext with all services
  4. Registers @server.list_tools() and @server.call_tool() closures
  5. Starts stdio_server() — blocks until Claude Code disconnects

router.py — Model Routing

Router.route() uses keyword matching on INTENT_KEYWORDS and STAKES_HIGH/STAKES_LOW lists to detect intent and stakes. The DEFAULT_TIERS dict maps intent → tier, with three overrides:

  • If intent is summarize and input > 100K tokens → upgrade to balanced
  • If intent is code and "multi-file" in text or tool chain depth ≥ 3 → upgrade to powerful
  • If intent is agent_loop and tool chain depth > 5 → upgrade to powerful

Stakes override: high forces powerful; low downgrades balanced → fast.

Budget guard: if monthly_budget_usd is provided and projected monthly spend exceeds 110% of budget, _downgrade_tier() reduces tier by one step.

compare_providers()

Iterates all configured providers and tiers, estimates input + output cost for the given text (using 20% of input tokens as output estimate), and returns a list sorted by total_cost_usd.

rewriter.py — Prompt Rewriting

Rewriter.__init__ builds self.preamble_phrases (lowercased list) and self.filler_words (set) from config at construction time — not per-call.

rewrite() pipeline:

  1. Count original tokens
  2. Strip matching preamble (first match only, from start of text)
  3. Remove filler words from all positions via word-by-word filter
  4. Prepend role prefix (if role has one)
  5. Count rewritten tokens, compute saving_pct
  6. Warn if rewritten text collapsed to <8 tokens (likely over-stripped)
Filler removal is word-boundary based (strips punctuation before comparison). It won't damage compound words or code identifiers.

optimizer.py — Context Optimization

Splits text into sentences (by .!? with minimum 10-char sentences). Scores each sentence using term frequency weighted by sentence position (earlier = higher weight). Greedily selects top-scored sentences until the token budget is reached.

Scoring weights:
position_weight = 1.0 / (1 + sentence_index)   # earlier sentences score higher
tf_score = sum(word_freq[w] for w in sentence_words) / len(sentence_words)
final_score = tf_score * position_weight

Returns OptimizeResult with optimized text, saving_pct, and chunks_dropped.

cache_planner.py — Cache Planning

CachePlanner.plan() iterates messages and assigns TTL based on role + label combinations (see User Guide). It then calls two cost calculators:

  • _calc_cost_without_cache() — standard input rate × total tokens × reuse count
  • _calc_cost_with_cache() — cache_write_5m rate for first write + cache_hit rate × (reuse-1) for cached tokens + standard rate for uncached

The 5-minute TTL is always used for cost calculation (conservative). The 1-hour TTL is available for stable content (system prompts, tool definitions).

batcher.py — Prompt Batching

Constructs a numbered task list with role framing. The savings come from eliminating per-task context-reload overhead tokens. Token savings are calculated as:

individual_tokens = sum(count_tokens(t) for t in tasks)
batched_tokens = count_tokens(batched_prompt)
saving_pct = (individual_tokens - batched_tokens) / individual_tokens * 100

summarizer.py — Thread Summarization

Uses the same sentence-scoring algorithm as optimizer.py but adds a reset_prompt generation step. The reset prompt is formatted as:

"Continue from: {summary}\nNext step: {last_sentence}"

where last_sentence is the last sentence from the original conversation (preserved to maintain continuity).

stats.py — Statistics & Cost Tracking

StatsService.record() calculates cost using calc_cost() and writes to the history table asynchronously. Errors are printed to stderr (not re-raised) so a failed stats write never breaks a tool call.

calc_cost() is a static method that handles two modes:

  • batch mode: uses batch_input_per_mtok / batch_output_per_mtok
  • standard mode: separates uncached input (input_tokens - cached_input_tokens) billed at full rate, cached input billed at cache_hit_per_mtok

session_manager.py — Session Tracking

Uses an async upsert on the sessions table (SQLite INSERT OR REPLACE). check_timeout() computes idle minutes from last_ping_ts and applies thresholds:

idle_minutes = (now - last_ping_ts).total_seconds() / 60

if idle_minutes >= idle_threshold: status = "expired"
elif idle_minutes >= warn_threshold: status = "warn"
else: status = "active"

tokenizer.py — Token Counting

Uses tiktoken's cl100k_base encoding (shared by all currently-configured models). The inflation_vs_baseline field in TokenizerSpec allows per-model adjustment (Opus 4.7 uses 1.20, meaning its tokens count 20% higher for the same text).

Falls back to len(text.split()) * 1.3 whitespace estimate if tiktoken is unavailable.

db.py — Database Migrations

Two migration sets applied idempotently at every startup. Each version is gated by a row in schema_version:

V1: Initial schema
  schema_version, history (id, ts, tool, model, tokens, cost, saving_pct)
  Indexes: idx_history_ts, idx_history_tool

V2: FinOps + sessions
  ALTER TABLE history ADD COLUMN duration_ms
  ALTER TABLE history ADD COLUMN project
  ALTER TABLE history ADD COLUMN team
  CREATE TABLE sessions (session_id, started_ts, last_ping_ts)

V2 migration catches duplicate column name errors to be idempotent (safe to re-run).

Adding a Migration (v3+)

  1. Increment SCHEMA_VERSION = 3 in db.py
  2. Define MIGRATIONS_V3 list of SQL strings
  3. Write async def _apply_v3(db) following the pattern of _apply_v2: check for existing version row, run migrations idempotently (catch duplicate column name for ALTERs), insert version row
  4. Call await _apply_v3(db) from init_db()
  5. Add a test in tests/test_db.py that runs migration on a fresh in-memory database

Adding a New Tool

Five touch points, all in server.py and the new service file:

  1. Create the service. Add src/promptwise/my_service.py. Return a frozen dataclass from types.py (add the dataclass there first). 
    class MyService:
    def __init__(self, config: AppConfig): ...
    def process(self, text: str) -> MyResult: ...
  2. Add to ServerContext. Add my_service: MyService field and construct it in main().
  3. Register in list_tools(). Add a Tool(...) entry with name, description, and inputSchema.
  4. Dispatch in call_tool(). Add an elif name == "my_tool": branch. Call the service, record to stats, return json.dumps({...}). 
    elif name == "my_tool":
    result = ctx.my_service.process(arguments.get("text", ""))
    await ctx.stats.record(tool="my_tool", saving_pct=result.saving_pct)
    return json.dumps({"output": result.output, "saving_pct": result.saving_pct})
  5. Write tests. Add tests/test_my_service.py with at least: happy path, edge case (empty input, over-budget input), and expected saving_pct bounds.
The call_tool() function wraps everything in a try/except and returns {"error": ..., "type": ..., "tool": ...} on any exception. You don't need per-service error handling — just let exceptions propagate to the top-level handler.

Adding a Provider

  1. Add model entries to pricing.yaml with the new provider's model IDs and rates. Set provider field to your new provider name.
  2. Add a provider block to providers.yaml with tiers.fast/balanced/powerful pointing to the model IDs you defined.
  3. Call reload_config (or restart). PromptWise will validate the cross-references.
  4. The new provider will appear automatically in compare_providers output.
pricing.yaml — add model:
my-model-fast:
  display_name: "MyProvider Fast"
  provider: myprovider
  tier: fast
  context_window: 200000
  max_output: 8192
  rates:
    input_per_mtok: 0.50
    ...

providers.yaml — add provider:
myprovider:
  display_name: "My Provider"
  aliases: [myp]
  tiers:
    fast: my-model-fast
    balanced: my-model-balanced
    powerful: my-model-powerful
  peak_hours_utc: []
  feature_warnings: []

Adding a Role

Add an entry to roles.yaml under roles::

legal:
  display_name: "Legal"
  prefix: "With precision and attention to regulatory detail, "
  description: "Contract review, compliance, risk analysis"

Then call reload_config. The new role will be accepted by rewrite_prompt immediately.

Updating Pricing

  1. Edit the relevant model's rates block in pricing.yaml.
  2. Update last_verified to today's date (YYYY-MM-DD).
  3. Call reload_config — no server restart needed.
Verify rates at the official pricing pages: Anthropic → anthropic.com/pricing, OpenAI → platform.openai.com/pricing, Gemini → ai.google.dev/pricing. Rates in the bundled pricing.yaml were last verified 2026-05-12.

Test Patterns

Config fixture

Most tests use a minimal config fixture rather than loading real YAML files:

@pytest.fixture
def config(tmp_path):
    # Write minimal valid YAML files to tmp_path
    (tmp_path / "pricing.yaml").write_text("""...""")
    (tmp_path / "providers.yaml").write_text("""...""")
    (tmp_path / "roles.yaml").write_text("""...""")
    return load_config(tmp_path)

Async DB tests

Stats and session tests use pytest-asyncio with a temporary DB path:

@pytest.mark.asyncio
async def test_record_and_snapshot(config, tmp_path):
    db_path = tmp_path / "test.db"
    await init_db(db_path)
    svc = StatsService(config, db_path)
    await svc.record(tool="route_request", model="claude-sonnet-4-6", input_tokens=100)
    snap = await svc.snapshot()
    assert snap.total_calls == 1

MCP Entry Point

The MCP server is launched by Claude Code using the mcpServers block in plugin.json:

{
  "mcpServers": {
    "promptwise": {
      "command": "python",
      "args": ["-m", "promptwise.server"],
      "cwd": "${pluginDir}"
    }
  }
}

${pluginDir} resolves to C:\test\PromptWise (the plugin install location). The server module is resolved from src/promptwise/server.py because src/ is on the Python path via the pyproject.toml package declaration.

The config directory is resolved at runtime as 2 levels up from __file__ (src/promptwise/server.py → src/ → project root), so pricing.yaml, providers.yaml, and roles.yaml are always loaded from the project root regardless of the working directory.

// 01 — What You Already Have

v2.0 Engine — Reuse vs Extend

v2.0 is a strong foundation. v3.0 does NOT rebuild it — it adds the skill layer on top. Every module below is preserved and extended, not replaced.

core/router.py
Intent detection, model selection (Opus/Sonnet/Haiku). Already routes by complexity.
✓ Reuse as-is
core/orchestrator.py
DAG-based multi-step execution with failure strategies. Will power skill chains in v3.
▲ Extend for skill DAGs
core/security.py
5-level preflight: syntax, secrets, destructive, supply chain, permissions.
▲ Extend with PII + injection
core/context_engine.py
Intelligent file selection via dependency graph + TF-IDF + LLM scoring.
✓ Reuse as-is
core/memory_manager.py
SQLite session persistence, 4-week retention, JSON export.
▲ Extend for skill state tracking
core/compression_engine.py
Caveman compression lite/full/ultra with 15–20% token savings.
✓ Reuse as-is
core/role_intelligence.py
6 org roles: IT/Dev/EM/PM/SM/NTM with adaptive routing.
▲ Map roles to skill access
plugins/plugin_system.py
Hybrid keyword + burn-rate detection, auto-load at confidence > 0.6.
▲ Extend: skills use same detection
integrations/mcp_adapters.py
Claude (native), Cursor (socket), Codex (REST), Gemini (gRPC) adapters.
✓ Reuse — already multi-platform
Key insight from your own roadmap: "v2.0 is a strong engine. The gap is a skill library (40+ skills) + skill composition layer on top." This plan builds exactly that — zero breaking changes to v1.2.0 or v2.0.0.

Skill Frontmatter Schema

Every skill is a Markdown file with YAML frontmatter. This schema is the contract all 40+ skills follow.

Field
Type / Values
Purpose
name
string (kebab-case)
Unique skill ID, used in invoke_skill and skill_chain tools

string (1-2 sentences)
Used by auto-trigger keyword matching and list_skills output

string[] (keywords/phrases)
Matched against user prompt by plugin_system.py detection engine

string[] (other skill names)
Skills that must run before this one. Orchestrator builds the DAG.

--- name: feature-dev description: "End-to-end feature development: brainstorm → TDD → code-review → verify" triggers: ["build feature", "implement feature", "new feature", "add feature"] depends_on: ["tdd", "code-review", "verification-before-completion"] output_schema: type: object properties: implementation_files: {type: array} test_files: {type: array} review_summary: {type: string} verified: {type: boolean} roles: [Dev, IT] model_tier: auto --- # Feature Development Skill ## System Prompt You are a senior software engineer executing a complete feature development cycle... ## Steps 1. Brainstorm and clarify requirements with the user 2. Write failing tests first (TDD) 3. Implement to pass tests 4. Run code-review skill on output 5. Run verification-before-completion skill

3 New MCP Tools (adds to existing 24)

Tool Name Input What It Does Claude Implementation
invoke_skill NEW skill_name, context, params Loads the skill markdown, injects context, sends to router.py for model selection, executes via orchestrator.py Claude reads the skill system prompt + user context. Routes complexity to Opus/Sonnet/Haiku. Returns typed output per output_schema.
list_skills NEW role?, category? Returns all available skills filtered by current user role and optional category. Shows triggers, model tier, dependencies. Scans skills/ directory, parses frontmatter, filters by role_intelligence.py current role. Returns structured JSON.
skill_chain NEW skills[], mode, context Runs multiple skills sequentially or in parallel. Validates typed handoffs between skills. Handles failures per orchestrator strategy. Orchestrator.py builds DAG from depends_on fields. Each skill output validated against next skill's input schema before handoff.
🤖 Claude Implementation — Phase 3.0
  • Create src/promptwise_v2/skills/ directory; each skill is a .md file with YAML frontmatter parsed by frontmatter Python library
  • Add SkillLoader class to plugin_system.py: scans skills/ on startup, indexes triggers into keyword matcher, auto-suggests at confidence >0.6
  • Extend orchestrator.py: add execute_skill_chain() method that topologically sorts depends_on DAG, handles sequential + parallel modes
  • Add output schema validation using jsonschema library — validate each skill output before passing to next in chain
  • Register 3 new tools in server.py alongside existing 24 — total becomes 27 after Phase 3.0
  • Use Claude's tool use API for complex skills that need multi-turn reasoning; simple skills use single completion

Dev Workflow Skills (8 skills)

Phase 3.1 Dev Workflow Skills
Requires: Phase 3.0
brainstorm → tdd → feature-dev → code-review → verification-before-completion → finishing-branch
tdd Dev · Sonnet
Enforces test-first development. Writes failing tests based on requirements, then guides implementation to pass them. Integrates with pytest/jest/go test.
# Claude prompt pattern system: "You are a TDD expert. NEVER write implementation before tests. Given requirements, write comprehensive failing tests first using {framework}. Then implement minimally to pass." tools: [read_file, write_file, run_terminal] model: sonnet # cost-effective for structured output
systematic-debugging Dev · Opus
Structured debugging: reproduce → isolate → hypothesize → verify. Uses context_engine.py to pull relevant files. Records hypothesis log in memory_manager.
# Multi-step Claude reasoning step1: Reproduce + capture error context step2: Isolate via binary search / bisect step3: Generate ranked hypotheses (Opus) step4: Verify each hypothesis with tests memory: store hypothesis log per session
code-review Dev/EM · Sonnet
Automated code review covering correctness, style, complexity, security signals, and test coverage. Outputs structured review JSON consumed by CI hooks.
# Routes to existing codereview_bridge.py output_schema: { issues: [{severity, file, line, message}], score: 0-100, approved: boolean, suggestions: string[] } # Feeds into finishing-branch skill
verification-before-completion Dev · Haiku
Checklist verification before marking any task done: tests pass, no TODOs left, docs updated, no console logs, security checks clear. Fast — uses Haiku.
# Haiku-optimized checklist runner checks: [ run_tests → must_pass, grep_todos → must_be_zero, check_docs → updated, security_check → clear # calls security.py ] blocks: finishing-branch if any fail
refactoring Dev · Sonnet
Systematic refactoring with safety net: writes characterization tests first, refactors incrementally, verifies tests still pass at each step. Uses context_engine for related files.
# Safe refactoring protocol phase1: write characterization tests phase2: identify code smells (complexity, duplication, coupling) phase3: refactor one concern at a time phase4: run tests after each change
git-workflow Dev · Haiku
Enforces git best practices: conventional commits, branch naming, PR description generation, merge conflict resolution guidance. Writes commit messages automatically.
# Auto-generates conventional commits commit_format: "type(scope): description" pr_template: Summary + Changes + Testing branch_naming: feat/fix/chore/docs prefixes model: haiku # fast, cheap, structured
🤖 Claude Implementation — Phase 3.1
  • All 8 skills are Markdown files in skills/dev/ — no new Python modules needed beyond Phase 3.0's SkillLoader
  • feature-dev skill uses skill_chain tool internally: chains brainstorm → tdd → code-review → verification automatically
  • systematic-debugging uses Opus with extended thinking (budget_tokens: 8000) for hypothesis generation — most complex reasoning task
  • verification-before-completion calls existing security_check MCP tool (v2 tool) as part of its checklist
  • git-workflow uses Claude's tool use to actually run git commands via the existing terminal tool in Cline/Claude Code

// 05 — Phase 3.3

Security Hardening (7 features)

Phase 3.3 Security Hardening
Extends existing security.py (5-level preflight)

Current security.py has 5 levels: syntax, secrets, destructive, supply_chain (stub), permissions. Phase 3.3 replaces the stub and adds 6 new checks — all run BEFORE any LLM call.

PII Detector All Roles · Haiku
Scans prompts AND outputs for PII before sending to LLM and before returning to user. Detects names, emails, phone numbers, SSNs, credit cards, addresses using regex + Claude.
# Runs in security.py preflight (level 6) scan_input: regex fast-path (emails, SSN, CC) scan_llm_output: Claude Haiku classification action: redact | warn | block (configurable) add_to_yaml: security.pii_detection: true
Prompt Injection Detector All Roles · Haiku
Detects jailbreak patterns, role-override attempts, and injection in user input before it reaches Claude. Pattern library + Haiku classifier for novel attempts.
# Pattern library + LLM classifier patterns: ignore-previous, DAN, act-as, forget-instructions, developer-mode llm_check: Haiku scores injection confidence threshold: block if confidence > 0.7 log: all attempts to monitoring.py
OWASP Top-10 Checker Dev/IT · Sonnet
Checks generated code for OWASP Top-10 vulnerabilities: SQLi, XSS, SSRF, IDOR, broken auth, etc. Runs as a post-generation hook before returning code to user.
# Post-generation security hook checks: A01-A10 OWASP 2021 categories claude_prompt: specialized security reviewer output: [{vuln, severity, line, fix}] blocks: HIGH severity before returning code
Real CVE Lookup Dev/IT · API
Replaces the current supply_chain stub with real OSV.dev API integration. Checks dependencies in requirements.txt, package.json, go.mod, pom.xml against live CVE database.
# Replaces stub in security.py api: https://api.osv.dev/v1/query reads: requirements.txt, package.json, go.mod, pom.xml, Cargo.toml claude: explains severity + suggests fix cache: SQLite (memory_manager), 24hr TTL
License Compliance IT/Legal · Haiku
Detects GPL contamination and license conflicts in dependencies. Classifies SPDX license identifiers and flags incompatibilities with project's declared license.
# SPDX license compatibility matrix matrix: MIT/Apache/BSD vs GPL/AGPL/LGPL source: license fields from package APIs claude: explains conflict + remediation output: {compatible: bool, risks: []} roles: [IT, NTM] # management visibility
SBOM Generator IT · Haiku
Generates a Software Bill of Materials in CycloneDX or SPDX JSON format from all dependency files. Feeds into CVE lookup and license compliance skills.
# CycloneDX 1.5 JSON output reads: all lock files (poetry.lock, yarn.lock) format: CycloneDX JSON or SPDX JSON claude: Haiku for component summarization chains_to: cve-lookup, license-compliance
🤖 Claude Implementation — Phase 3.3
  • PII Detector and Prompt Injection Detector hook into security.py as levels 6 and 7 — runs on every request, zero config needed
  • OSV.dev API is free and public — use httpx async client, cache results in memory_manager SQLite to avoid rate limits
  • OWASP checker prompt: give Claude a specialized "security auditor" system prompt with OWASP cheat sheet embedded in skill markdown
  • New YAML config keys: security.pii_detection: true, security.injection_detection: true, security.owasp_check: post_generation
  • SBOM generator uses existing dependency file detection from context_engine.py to locate lock files
// 06 — Phase 3.4

AI/Prompt Engineering Skills (7 skills)

Phase 3.4 AI / Prompt Engineering Skills
Unique differentiator — no other plugin covers this
SkillWhat it doesClaude Implementation Notes
prompt-registry Versioned SQLite store for prompts. Save, version, tag, search, and compare prompts. Team-shared or personal. Extends memory_manager.py with a prompts table. Claude Haiku generates embeddings for semantic search. Exposed via new MCP tool manage_prompts.
multi-model-eval A/B test the same prompt across Opus/Sonnet/Haiku (or any configured models). Returns quality scores, cost, and latency comparison. Sends prompt in parallel to all 3 tiers via router.py. Claude Opus acts as judge: scores each response on accuracy, completeness, format. Returns comparison table + cost breakdown.
rag-optimizer Analyzes RAG pipeline quality: chunk size, overlap, embedding model, retrieval strategy. Suggests optimizations with before/after comparison. Claude reviews RAG config + sample retrieved chunks. Evaluates context relevance, identifies retrieval failures. Uses context_engine.py internals as example. Opus-level analysis.
few-shot-builder Builds optimized few-shot example sets for any Claude prompt. Selects diverse, high-quality examples that maximize in-context learning. Claude analyzes existing examples in prompt-registry, scores diversity + coverage. Generates synthetic examples to fill gaps. Output: ordered few-shot array ready to paste.
agent-chain-designer Visual + textual designer for multi-agent workflows. Describe a complex task, get a recommended agent chain with skill dependencies mapped. Opus maps task → skill DAG → agent roles. Outputs skill_chain() call ready to execute. Integrates with orchestrator.py's DAG builder. Generates Mermaid flowchart of the chain.
system-prompt-auditor Reviews existing system prompts for issues: ambiguity, conflicting instructions, missing edge cases, jailbreak susceptibility, excessive restriction. Claude acts as adversarial evaluator. Red-teams the system prompt, identifies weaknesses, suggests improvements. Also checks against injection-detector patterns from Phase 3.3.
model-migration-advisor When switching models (e.g., GPT-4 → Claude, Opus → Sonnet), analyzes prompt compatibility, identifies capability gaps, suggests adaptation strategies. Uses router.py's model knowledge + multi-model-eval to test critical prompts on target model. Flags incompatibilities, generates migration checklist. References existing mcp_adapters.py model configs.
🤖 Claude Implementation — Phase 3.4
  • multi-model-eval uses Claude's API directly to run parallel completions — use asyncio.gather() in Python for true parallelism across Opus/Sonnet/Haiku
  • Claude-as-judge pattern for multi-model-eval: Opus scores outputs with a structured rubric prompt; returns JSON scores per criterion
  • prompt-registry adds 2 new MCP tools: save_prompt(name, content, tags) and search_prompts(query) — builds on existing memory_manager SQLite
  • agent-chain-designer is the most powerful Phase 3.4 skill — it makes PromptWise self-aware of its own skill system; uses Opus with full skill catalog in context
  • system-prompt-auditor should use Claude's extended thinking for adversarial red-teaming (budget_tokens: 5000) to find non-obvious weaknesses

Generates comprehensive test suites for existing code. Supports pytest, Jest, and Go test. Reads source via context_engine, infers edge cases, generates both happy path and failure tests.
# Framework detection + test gen detect: pytest/jest/go via project files reads: source file via context_engine.py generates: unit + integration + edge cases style: Arrange-Act-Assert pattern enforced

Reads coverage reports (pytest-cov, Istanbul, go cover) and identifies the highest-value uncovered code paths. Prioritizes by risk: complex/critical code first.
# Coverage gap analysis reads: coverage.xml, lcov.info, cover.out ranks: uncovered paths by cyclomatic complexity output: [{file, function, priority, reason}] chains_to: test-generator for gap filling
api-testing Dev/IT · Sonnet
Generates API test suites from OpenAPI specs or by scanning route definitions. Creates positive, negative, and boundary tests. Outputs Postman collections or pytest files.
# OpenAPI → test suite input: OpenAPI spec (from api-docs skill) OR route files (from context_engine) generates: 200/4xx/5xx cases per endpoint output: pytest + Postman JSON collection
e2e-test-designer Dev · Opus
Routes to existing playwright_bridge.py for frontend detection. Designs E2E test scenarios covering critical user journeys. Generates Playwright test code with Page Object Model pattern.
# Routes to playwright_bridge.py (existing) detects: frontend via playwright_bridge designs: user journey scenarios (Opus) generates: Playwright tests with POM pattern validates: against existing playwright_bridge
test-data-generator Dev · Sonnet
Generates realistic, schema-consistent test data. Reads DB schemas or type definitions, produces fixtures in JSON/CSV/SQL. Includes edge cases: null, empty, max-length, unicode.
# Schema-aware test data gen reads: SQLAlchemy models, TypeScript types, JSON Schema, Pydantic models generates: happy + edge + boundary cases formats: JSON fixtures, CSV, SQL INSERT pii_safe: uses fake data (Faker library)
🤖 Claude Implementation — Phase 3.5
  • test-generator integrates with tdd skill from Phase 3.1: tdd writes failing tests first, test-generator fills coverage gaps afterwards
  • e2e-test-designer reuses playwright_bridge.py (already exists) — skill is thin wrapper that adds Opus-level scenario design before bridge execution
  • test-data-generator uses Python Faker library for PII-safe realistic data; Claude handles schema parsing and edge case identification
  • api-testing skill chains with api-docs skill from Phase 3.2: api-docs generates the spec, api-testing consumes it
